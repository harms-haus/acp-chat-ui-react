# Architecture: @acp/chat-core

## What This Package Does

`@acp/chat-core` is the TypeScript engine that sits between a Rust bridge server and a frontend UI. Its job: take raw WebSocket messages from an ACP (Agent Communication Protocol) agent, parse them, and turn them into data structures a React (or Svelte) UI can actually render.

The package handles five concerns, and it handles them without depending on any UI framework:

1. **Wire protocol** - parsing versioned envelopes from the Rust bridge
2. **Transport** - managing the WebSocket lifecycle (connect, reconnect, disconnect)
3. **Session control** - sending JSON-RPC requests to the agent and routing responses
4. **State normalization** - converting raw ACP updates into flat, queryable data structures
5. **UI helpers** - pure functions for composer logic and thought grouping

Everything else the package exports exists to support one of those five things.

---

## Layered Architecture

The package is organized into layers. Each layer only depends on the layers below it, never the other way around.

```
+---------------------------------------------------+
|  UI Layer (React / Svelte, not in this package)   |
+---------------------------------------------------+
           consumes
+---------------------------------------------------+
|  Helpers                                           |
|  composer-logic.ts, thought-stack-logic.ts         |
+---------------------------------------------------+
           reads from
+---------------------------------------------------+
|  Normalization                                     |
|  store.ts                                          |
+---------------------------------------------------+
           fed by
+---------------------------------------------------+
|  Session                                           |
|  controller.ts, replay-controller.ts               |
+---------------------------------------------------+
           uses
+---------------------------------------------------+
|  Bridge        |  Transport                        |
|  parser.ts     |  client.ts                        |
+---------------------------------------------------+
           depends on
+---------------------------------------------------+
|  Generated Types (from Rust via ts-rs)             |
|  BridgeEnvelope, BridgeStatus, ...                 |
+---------------------------------------------------+
```

Data flows downward through the stack. Generated types have zero dependencies. The bridge and transport layers use those types. The session layer uses the bridge and transport. Normalization consumes events from the session layer. Helpers are pure functions that read normalized state.

## Module Breakdown

### Generated Types

**Directory:** `src/generated/`

These files are auto-generated by [ts-rs](https://github.com/Aleph-Alpha/ts-rs) from Rust type definitions. **Do not edit them by hand.** If you need to change a generated type, change the Rust source and regenerate.

The key generated types:

| Type | File | What it represents |
|---|---|---|
| `BridgeEnvelope` | `BridgeEnvelope.ts` | The versioned wrapper for every message from the bridge. Carries a version number, sequence counter, timestamp, and a discriminated union payload (`acp_payload`, `bridge_status`, `stderr`, `process_exit`, `replay_metadata`, `start_agent`). |
| `BridgeMessage` | `BridgeMessage.ts` | The inner message variants, same discriminated union without the envelope metadata. |
| `BridgeStatus` | `BridgeStatus.ts` | Bridge lifecycle states: `starting`, `connected`, `reconnecting`, `disconnected`, `error`. |
| `UnsupportedVersionError` | `UnsupportedVersionError.ts` | Structured error when an envelope arrives with an unrecognized version number. |

The `BridgeEnvelope` type is the single source of truth for the wire format. The Rust bridge never interprets ACP payload contents. It just wraps them in this envelope and sends them over the WebSocket.

---

### Bridge (Parser)

**Files:** `src/bridge/parser.ts`, `src/bridge/index.ts`

The bridge module validates and parses raw JSON strings into typed `BridgeEnvelope` objects. This is the first thing that happens to every incoming WebSocket message.

**Why this exists as a separate layer:** Version validation needs to happen before any downstream code touches the envelope. If the bridge server is upgraded and starts sending a new envelope version, the parser catches that immediately and returns a `BridgeVersionError` rather than letting malformed data propagate through the session and normalization layers.

Key exports:

- `parseEnvelope(json)` - parses a JSON string, validates the version, returns a typed `BridgeEnvelope`. Throws `BridgeVersionError` on version mismatch.
- `parseEnvelopeSafe(json)` - same as above but catches `BridgeVersionError` and returns it as a value instead of throwing.
- `ENVELOPE_VERSION` / `SUPPORTED_VERSIONS` - constants for the current protocol version (currently `1`).
- `validateEnvelope(envelope)` - validates an already-parsed object.

---

### Transport

**Files:** `src/transport/client.ts`, `src/transport/index.ts`

`TransportClient` manages the raw WebSocket connection. It handles:

- **Connection lifecycle** - `connect()`, `disconnect()`, and connection state tracking
- **Automatic reconnection** - exponential backoff with configurable max attempts and delay caps
- **Message routing** - incoming messages are parsed as envelopes and emitted to subscribers, or routed as init responses
- **Init protocol** - `initLive()` and `initReplay()` for establishing the session mode with the bridge server

**Why reconnection lives here:** The transport layer is the only place that knows about the WebSocket itself. Reconnect logic with exponential backoff belongs at this level because higher layers (session controller) don't need to know whether a message arrived on the first connection or the fifth reconnection attempt.

The client uses a typed event emitter pattern. You subscribe with `on("envelope", handler)` and get back an unsubscribe function. Three events are available:

| Event | Payload | When it fires |
|---|---|---|
| `statusChange` | `ConnectionStatus` | Connection state transitions (`disconnected`, `connecting`, `connected`, `reconnecting`, `error`) |
| `envelope` | `BridgeEnvelope` | Every successfully parsed incoming envelope |
| `error` | `Error` | Parse failures, version mismatches, WebSocket errors |

---

### Session

**Files:** `src/session/controller.ts`, `src/session/replay-controller.ts`, `src/session/capture-interceptor.ts`

The session layer is where the ACP protocol lives. It wraps the transport and adds JSON-RPC request/response semantics on top of the raw envelope stream.

#### SessionController

The main controller for live sessions. It:

- Sends JSON-RPC requests (`initialize`, `session/new`, `session/prompt`, `session/load`, `session/list`) and matches responses by request ID
- Handles timeouts on pending requests (default 30 seconds)
- Parses `acp_payload` envelopes to extract JSON-RPC responses, `session/update` notifications, and `session/request_permission` notifications
- Emits typed events that the normalization layer consumes

**Events emitted by SessionController:**

| Event | When |
|---|---|
| `statusChange` | Connection or bridge status changes |
| `sessionUpdate` | Every ACP update (messages, thoughts, tool calls, permissions) |
| `traffic` | Every message in both directions (for debugging and capture) |
| `error` | Any error from transport or request timeouts |
| `sessionClearing` | Before loading a different session (signals UI to reset state) |
| `permissionRequest` | Agent asks for user approval to run a tool |

#### ReplayController

A drop-in alternative to `SessionController` that drives the bridge in replay mode. It shares the same event interface so the UI layer can swap between live and replay without changing any subscription code.

Key differences from `SessionController`:

- Uses `initReplay(script, sessionId)` instead of `initLive()` to start
- Exposes `setReplaySpeed()` for controlling playback speed
- Carries fake `modes` and `models` in its state (so settings panels can render meaningful options)
- Handles `replay_metadata` envelopes from the bridge
- Reconnect is disabled (replay is one-shot)

#### DefaultSessionCaptureInterceptor

Wraps a `SessionController` and records every event that flows through it. This is how session replays are created: you start a capture, interact with the agent, then export the captured session as a JSONL file.

The interceptor listens to `traffic` and `sessionUpdate` events, timestamps each one, estimates token counts, and extracts metadata (which modes and models were active). When capture stops, it writes two files to `fixtures/replay-data/captured/`:

- `session-data.json` - metadata (session ID, timestamps, token totals, modes, models)
- `replay-events.jsonl` - one JSON object per line, each wrapping a `BridgeEnvelope`

---

### Normalization

**Files:** `src/normalization/store.ts`, `src/normalization/index.ts`

The normalization layer converts raw ACP session updates into flat, queryable data structures that a UI can render without further processing.

**Why normalization exists:** ACP agents send updates as a stream of incremental chunks. An agent message might arrive as five separate `agent_message_chunk` events, each appending a few words. The normalization layer accumulates these chunks into a single `NormalizedMessage` with the full text and a status of `streaming` or `complete`. Without this layer, every UI component would need its own chunk-accumulation logic.

The core type is `NormalizedState`, which holds:

- `messages` - a `Map<string, NormalizedMessage>` keyed by message ID
- `thoughts` - a `Map<string, NormalizedThought>` keyed by thought ID
- `toolCalls` - a `Map<string, NormalizedToolCall>` keyed by tool call ID
- `permissionRequests` - a `Map<number, NormalizedPermissionRequest>` keyed by request ID
- `timelineOrder` - an ordered array tracking the sequence all items appeared in
- `turnIdToMessageId` - a lookup map for finding messages by their turn ID

`applySessionUpdate(state, params)` is the main entry point. It dispatches on the update type:

| Update type | What it does |
|---|---|
| `agent_message_chunk` | Creates or appends to an agent message |
| `agent_thought_chunk` | Creates or appends to a thought |
| `user_message` / `user_message_chunk` | Creates a user message (no streaming for user messages) |
| `tool_call` / `tool_call_update` | Creates or updates a tool call record |
| `permission_request` | Creates a permission request for user approval |

Query functions (`getMessages`, `getTimeline`, `getToolCalls`, etc.) read from the state and return sorted, filtered arrays. The `getTimeline()` function is particularly important: it returns items in the order they appeared, which is how the chat UI renders its message list.

---

### Presets (Configuration)

**Files:** `src/presets/launch.ts`, `src/presets/index.ts`

The presets module reads environment variables and produces a typed `LaunchPreset` object that tells the app how to start. This is how the app knows whether to connect to a live agent or replay a recorded session.

Environment variables consumed:

| Variable | Purpose |
|---|---|
| `ACP_LAUNCH_CMD` | Command to start the agent (proxy mode) |
| `ACP_SESSION_ID` | Pre-existing session to load |
| `ACP_CWD` | Working directory for the agent |
| `ACP_BRIDGE_MODE` | `"proxy"` for live, `"replay"` for recorded |
| `ACP_AUTO_CONNECT` | `"true"` or `"1"` to connect on startup |
| `ACP_BRIDGE_URL` | WebSocket URL (default: `ws://127.0.0.1:8765`) |
| `ACP_REPLAY_FILE` | Path to the replay file (replay mode) |

`parseLaunchPreset()` reads these and returns a `LaunchPreset`. `isPresetValid(preset)` checks that required fields are present for the chosen mode.

---

### Replay Types

**File:** `src/replay/types.ts`

Type definitions for the replay system. Defines the shape of recorded session data (`ReplaySessionMetadata`, `ReplaySessionData`, `ReplayEvent`, `ReplayManifest`) and a token estimation function (`estimateTokenCount`, which uses the rough approximation of `text.length / 4`).

This module is a dependency of both the capture interceptor (which writes replay data) and the replay controller (which reads it).

---

### Helpers

**Files:** `src/helpers/composer-logic.ts`, `src/helpers/thought-stack-logic.ts`

Pure functions with zero side effects and zero dependencies (other than normalization types). These were moved here from the Svelte package so both React and Svelte UIs can share them.

#### composer-logic.ts

Functions for the message input area:

- `shouldSendOnKeydown(key, shiftKey, isComposing)` - whether Enter should send the message
- `canSend(state)` - whether the send button should be active
- `isPromptActive(state)` - check if a prompt is in flight
- `shouldShowStopButton(lifecycle, lastItem)` - whether to show the stop button

Also defines `PromptLifecycleState` which tracks a prompt through `idle -> active -> complete | cancelled | error`.

#### thought-stack-logic.ts

Functions for grouping consecutive thoughts and tool calls into collapsible "thought groups" in the UI:

- `groupThoughtItems(timeline)` - groups adjacent thoughts/tool-calls into `ThoughtGroup` arrays
- `createGroupedTimeline(timeline)` - converts a flat timeline into alternating `message` and `thought_group` items
- `isThoughtGroupActive(groups, isAgentTyping)` - whether the last thought group is still active
- `shouldThoughtGroupBeOpen(...)` - collapse/expand logic for thought groups

## Data Flow

### Live Session

```
User types a prompt
        |
        v
+------------------+     JSON-RPC       +------------------+
| SessionController | ---"session/prompt"--> |  Rust Bridge    |
+------------------+                      +------------------+
        ^                                         |
        |         BridgeEnvelope                  |
        |     (acp_payload / bridge_status)       |
        |                                         v
        |                                   +-----------+
        +-----------------------------------|  ACP Agent |
                                            +-----------+

SessionController receives envelope
        |
        v
handleEnvelope() routes by type:
  - bridge_status  -->  update state, emit statusChange
  - acp_payload    -->  parse JSON-RPC, emit sessionUpdate
        |
        v
Normalization layer: applySessionUpdate()
  - agent_message_chunk --> accumulate text in NormalizedMessage
  - agent_thought_chunk --> accumulate text in NormalizedThought
  - tool_call            --> create/update NormalizedToolCall
  - permission_request   --> create NormalizedPermissionRequest
        |
        v
NormalizedState ready for UI rendering
```

### Replay Session

```
ReplayController.initReplay(script, sessionId)
        |
        v
TransportClient sends init command to bridge
        |
        v
Bridge reads .jsonl file, replays BridgeEnvelopes
        |
        v
ReplayController receives envelopes
  (same event interface as SessionController)
        |
        v
Normalization layer (same path as live)
        |
        v
NormalizedState ready for UI rendering
```

### Session Capture

```
DefaultSessionCaptureInterceptor
        |
        +-- wraps SessionController
        +-- listens on "traffic" and "sessionUpdate"
        |
        v
For every event:
  - record envelope, direction, timestamp, estimated tokens
  - extract mode and model metadata
        |
        v
On stopCaptureAndExport():
  - write session-data.json
  - write replay-events.jsonl
```

## Key Design Patterns

### Typed Event Emitter

Both `TransportClient` and `SessionController` use the same pattern: typed `on()` / `off()` methods that accept a string event name and a handler, returning an unsubscribe function. This avoids the need for a formal EventEmitter library while keeping the API type-safe.

```typescript
const unsub = controller.on("sessionUpdate", (params) => {
  // handle update
});
// later:
unsub(); // clean up
```

### Discriminated Union Envelope

`BridgeEnvelope` uses a discriminated union on the `type` field. Each variant has its own set of fields. This lets TypeScript narrow the type when you switch on `envelope.type`, and it makes it impossible to access the wrong fields for a given message type.

### Incremental State Accumulation

The normalization layer doesn't replace state on each update. It accumulates. An `agent_message_chunk` with a matching `turnId` appends its text to the existing message rather than creating a new one. This is why the UI can render a growing message without flickering: the same `NormalizedMessage` object gets updated in place.

### Controller Interchangeability

`SessionController` and `ReplayController` expose the same event interface (`statusChange`, `sessionUpdate`, `traffic`, `error`, `sessionClearing`, `permissionRequest`). The normalization and UI layers don't know or care which one they're connected to. This is how the app seamlessly switches between live and replay modes.

### Pure Helper Functions

The helpers module contains only pure functions. No state, no side effects, no framework dependencies. They take data in and return data out. This makes them trivially testable and usable from any framework.

## Important Notes

### Generated Types Must Not Be Edited

Everything in `src/generated/` is produced by `ts-rs` from Rust definitions. Manual edits will be overwritten on the next generation. If you need a type change, update the Rust source in the bridge crate and regenerate.

### No Runtime Dependencies

`@acp/chat-core` has zero production dependencies. It only uses TypeScript, and its devDependencies are `typescript` and `vitest`. This is intentional: the package should be loadable in any JavaScript environment without pulling in a dependency tree.

### Message IDs Are Generated Client-Side

IDs for messages (`msg_...`), thoughts (`thought_...`), and tool calls (`call_...`) are generated in the normalization layer, not by the agent. This is because ACP updates identify messages by `turnId`, but the normalized state needs stable IDs for React keys and DOM reconciliation. The ID format includes a timestamp and random suffix to avoid collisions.

### The Bridge Is a Passthrough

The Rust bridge does not interpret ACP payload contents. It wraps them in a `BridgeEnvelope` and sends them over the wire unchanged. All ACP protocol parsing (JSON-RPC method dispatch, update type detection) happens in the TypeScript session layer.
