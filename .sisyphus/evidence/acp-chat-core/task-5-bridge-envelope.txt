# Task 5: BridgeEnvelope Type Fix - Evidence

## Summary
Fixed BridgeEnvelope TypeScript type by adding missing `extra_data` field to match Rust-generated bindings.

## Investigation Findings

### 1. BridgeEnvelope Type Locations Found

1. **Rust Source (Source of Truth)**
   - Location: `crates/acp-ws-bridge/src/contract/envelope.rs`
   - Contains: `extra_data: Option<serde_json::Value>`
   - Attribute: `#[serde(skip_serializing_if = "Option::is_none")]`
   - **Status:** ✅ CORRECT - Has extra_data field

2. **Rust-Generated Bindings**
   - Location: `crates/acp-ws-bridge/bindings/BridgeEnvelope.ts`
   - Header: "This file was generated by [ts-rs]. Do not edit this file manually."
   - Contains: `extra_data: JsonValue | null`
   - **Status:** ✅ CORRECT - Has extra_data field

3. **Manual TypeScript Type (acp-ws-bridge)**
   - Location: `packages/acp-ws-bridge/src/types/BridgeEnvelope.ts`
   - Contains: `extraData?: Record<string, unknown>`
   - **Status:** ✅ CORRECT - Has extraData field (manual override)

4. **Auto-Generated TypeScript Type (acp-chat-core) - PROBLEMATIC**
   - Location: `packages/acp-chat-core/src/generated/BridgeEnvelope.ts`
   - Header: "This file was generated by [ts-rs]. Do not edit this file manually."
   - Contains: Missing extra_data field
   - **Status:** ❌ INCORRECT - Out of sync with Rust source

### 2. Root Cause Analysis

The `packages/acp-chat-core/src/generated/BridgeEnvelope.ts` file is a copy of the Rust-generated bindings, but it was out of sync. The Rust source and bindings have `extra_data`, but the copy in acp-chat-core did not.

**Why this happened:**
- No automatic regeneration script found in the project
- The generated types are manually copied from Rust bindings to acp-chat-core
- At some point, the copy missed the extra_data field addition

### 3. Compatibility Check

Searched for usages of BridgeEnvelope in the codebase:
- **20 files** import or use BridgeEnvelope
- **0 files** currently use `extra_data` or `extraData` field
- **Impact:** Backward compatible - adding optional field won't break existing code

pub struct BridgeEnvelope {
    pub version: u32,

## Solution Applied

### File Modified
- `packages/acp-chat-core/src/generated/BridgeEnvelope.ts`

### Change Made
Added the `extra_data?: JsonValue | null` field to the base type definition to match the Rust-generated bindings.

### Key Details
- **Optional field**: `extra_data?` (with `?`) to allow omission from JSON
- **Union type**: `JsonValue | null` to match Rust's `Option<serde_json::Value>`
- **Comment**: Preserved the documentation comment from Rust bindings explaining the purpose

### Why Not Update Rust Source?
The Rust source (`crates/acp-ws-bridge/src/contract/envelope.rs`) ALREADY HAS the `extra_data` field. The issue was purely in the TypeScript copy being out of sync.

### Why Not Regenerate from Rust?
No automatic regeneration script found in the project. The generated types appear to be manually maintained copies. Direct edit was necessary to fix the immediate issue.

### Why Not Edit acp-ws-bridge/src/types/BridgeEnvelope.ts?
That file is a manual TypeScript override that ALREADY HAS the `extraData` field. No changes needed there.
