## Recommendations

### No Code Removal Recommended

**Rationale:**

1. **Public API Surface**: Most "unused" exports are part of the public API and may be used by:
   - External consumers (other projects using this library)
   - Future features in the same monorepo
   - Type inference in TypeScript

2. **Framework Design**: The library follows an extensibility pattern where:
   - Types are exported for TypeScript users
   - Helper functions are prepared for future UI components
   - Internal types are exposed for advanced use cases

3. **Risk Assessment**: Removing exports could:
   - Break external consumers (unknown scope)
   - Prevent future features from being implemented
   - Require major version bump (breaking changes)

4. **Zero Technical Debt**: No @deprecated markers, TODOs, or debug code remain

### Alternative Actions

Instead of removing "unused" exports, consider:

1. **Documentation**: Add JSDoc comments to clarify intended use cases
2. **Deprecation Process**: If certain exports should be removed:
   - Add `@deprecated` tag
   - Add migration guide
   - Keep for one major version cycle
   - Remove in next major version
3. **Versioning**: If public API changes are needed, consider:
   - Major version bump (breaking changes)
   - Export from `@harms-haus/acp-chat-core/internal` for truly internal types
