## Rust Type Definition

### File: `crates/acp-bridge/src/contract/envelope.rs`

```rust
use serde_json::Value;

#[derive(Debug, Clone, Serialize, Deserialize, TS)]
#[ts(export)]
pub struct BridgeEnvelope {
    /// Envelope format version. Must be one of SUPPORTED_VERSIONS.
    pub version: u32,

    /// Sequence number for ordering messages in replay mode.
    /// Zero in live mode; monotonically increasing in replay mode.
    #[ts(type = "number")]
    pub seq: u64,

    /// Unix timestamp in milliseconds when the envelope was created.
    #[ts(type = "number")]
    pub timestamp_ms: u64,

    /// Optional free-form metadata. The ws-bridge treats this as opaque JSON.
    /// Specific interpretations (e.g., replay-speed) happen at the harness-server layer.
    #[serde(skip_serializing_if = "Option::is_none")]
    pub extra_data: Option<Value>,

    /// The message payload.
    #[serde(flatten)]
    pub message: BridgeMessage,
}
```

### Key Design Decisions:

1. **Type**: `Option<serde_json::Value>` - Maximum flexibility for arbitrary JSON
2. **Serialization**: `#[serde(skip_serializing_if = "Option::is_none")]` - Omit when None
3. **Validation**: None - ws-bridge treats extra_data as opaque JSON
4. **Semantics**: Harness-server defines meaning (e.g., replay-speed), not ws-bridge

## Implementation Notes

### ws-bridge (Rust)
- ✅ Serialize/deserialize as opaque JSON
- ✅ Preserve during message forwarding
- ✅ Never validate content
- ✅ Omit from serialization when None

### Harness-Server
- ✅ Define semantics (e.g., replay-speed)
- ✅ Validate content if needed
- ✅ Interpret for replay logic
- ✅ Attach to envelopes during replay

### React Layer (TypeScript)
- ✅ Receive in envelopes
- ✅ Forward to harness-server if needed
- ✅ Never assume specific structure
- ✅ Handle missing extraData gracefully

## Design Principles

1. **Free-form JSON**: No schema enforcement at ws-bridge level
2. **Optional**: Messages without extra-data are fully valid
3. **Preservation**: ws-bridge preserves but never interprets
4. **Harness-server semantics**: Specific meanings defined at higher layer
